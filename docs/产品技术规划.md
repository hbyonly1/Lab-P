# 实验报告智能处理平台产品与技术规划

## 1. 项目目标

将现有本地自动化工具升级为一个成熟的网站服务，面向学生提供实验报告图片识别、数据纠错、报告填空辅助和自动化填报服务。

当前已有能力包括：

- 收集学生手写实验报告图片
- 使用 AI 将图片识别为结构化实验数据
- 基于实验配置和 DOM 节点表，将数据填入学校内网实验报告系统
- 使用 Playwright / userscript 完成部分自动化流程
- 人工检查和纠错识别结果

新平台的目标是把这些能力产品化、流程化和可运营化，让用户可以按需求选择不同服务模式，并形成不同收费档位。

## 2. 产品模式

### 2.1 工具辅助模式

工具辅助模式面向希望节省主要操作时间、但愿意自己完成识别结果纠错和固定内容填写的用户。

用户流程：

1. 注册或登录平台
2. 选择实验项目
3. 输入学号和必要登录信息
4. 上传实验报告图片
5. 系统调用 AI 识别图片中的数据
6. 用户在网页中查看和纠正识别结果
7. 用户自行补充每个实验的固定填空和实验问题答案
8. 后台使用自动化工具登录学校实验报告系统
9. 按 DOM 节点表自动填入用户确认后的内容
10. 自动提交实验报告
11. 保存截图、日志和任务状态

平台负责：

- 图片上传
- AI 识别
- 结构化数据生成
- 在线纠错编辑器
- 基础格式校验
- 自动登录学校内网
- 自动填报实验报告
- 自动提交实验报告
- 截图留痕
- 任务状态通知

不负责：

- 人工代查、代改、代填
- 人工纠错
- 每个实验的固定填空
- 实验问题答案生成

适合场景：

- 用户愿意自己检查和修正 AI 识别结果
- 用户愿意自己完成固定填空和实验问题答案
- 用户希望平台代登录、代填报和自动提交
- 对价格敏感

建议定价：

- 单次按实验收费
- 或购买自动填报次数包
- 价格低于完整模式，但高于纯识别工具

示例：

```text
工具辅助模式：中低价
包含：图片识别 + 用户自行纠错 + 自动登录 + 自动填报 + 自动提交
不包含：人工审核 + 人工代改 + 固定填空代写 + 实验问题生成
```

### 2.2 完整模式

完整模式面向希望把流程交给后台处理的用户。

用户流程：

1. 注册或登录平台
2. 选择实验项目
3. 输入学号和必要登录信息
4. 上传实验报告图片
5. 提交完整处理任务
6. 后台 AI 识别图片
7. 管理员或审核员进行人工纠错
8. 后台使用自动化工具登录学校实验报告系统
9. 按 DOM 节点表自动填入正确位置
10. 保存截图、日志和任务状态
11. 用户查看处理结果

平台负责：

- 图片上传
- AI 识别
- 人工纠错
- 固定填空处理
- 实验问题答案生成
- 自动登录和自动填报
- 自动提交
- 截图留痕
- 异常处理
- 任务状态通知

完整模式收费应明显高于工具辅助模式，因为它包含人工纠错、固定填空处理、实验问题答案生成和异常兜底成本。

## 3. 模式对比

| 能力 | 工具辅助模式 | 完整模式 |
|---|---|---|
| 上传实验图片 | 支持 | 支持 |
| AI 图片识别 | 支持 | 支持 |
| 在线纠错 | 用户自己纠错 | 后台人工纠错 |
| 固定填空 | 用户自己填写 | 后台处理 |
| 实验问题生成 | 用户自己完成 | 后台处理 |
| 自动填入学校系统 | 支持 | 支持 |
| 自动提交 | 支持 | 支持 |
| 人工服务 | 不包含 | 包含 |
| 账号密码处理 | 需要 | 需要 |
| 收费 | 中低 | 较高 |
| 风险 | 中等 | 较高 |

## 4. 推荐技术栈

推荐采用以下技术栈：

```text
前端：Next.js + React + TypeScript
后端 API：FastAPI + Python
任务队列：Redis + Celery / Dramatiq / RQ
数据库：PostgreSQL
文件存储：MinIO / S3 / 阿里云 OSS / 腾讯 COS
浏览器自动化：Playwright Python
AI 识别：豆包 / 火山 Ark API
部署：Docker Compose 起步，后期可迁移到云服务器或内网服务器
```

选择理由：

- 当前核心代码大量使用 Python，后端继续使用 FastAPI 可以最大化复用现有逻辑
- Playwright Python 可以复用现有学校内网页面自动化方案
- Next.js 适合构建上传页面、纠错页面、任务状态页面和管理后台
- PostgreSQL 适合存储用户、任务、实验配置和结构化识别结果
- PostgreSQL 的 JSONB 适合保存 DOM 节点表、实验配置、AI 识别结果等半结构化数据
- Redis + Worker 适合处理 AI 识别、图片处理、自动填报等耗时任务
- MinIO 或对象存储适合保存大量实验图片、截图和日志附件

### 5.1 前端页面与角色权限设计

平台前端按角色展示不同页面，但权限不能只靠前端隐藏按钮控制。所有后端接口都必须检查当前登录用户的 `role`、任务归属和订单支付状态，防止用户通过直接请求接口越权访问。

角色建议：

```text
student  学生用户
reviewer 人工审核员 / 纠错员
admin    管理员
```

### student 可见页面

学生只能看到自己的任务、自己的订单和自己的结果。

页面：

- 登录页
- 引导页（用虚拟数据展示功能）
- 实验上传页（选择要填写的实验，上传图片，填写固定空和实验问题）
- AI 识别结果查看与自助纠错
- 结果页

student 允许操作：

- 使用学号和密码登录
- 选择实验
- 选择服务模式
- 创建订单
- 查看自己的支付状态
- 上传自己的实验图片
- 查看和修改自己的 AI 识别结果
- 在工具辅助模式下填写固定填空和实验问题答案
- 确认进入自动填报流程
- 查看自己的任务状态、失败原因、截图和结果

student 禁止操作：

- 查看其他学生任务
- 查看其他学生上传图片
- 修改其他学生识别结果
- 修改订单价格或支付状态
- 手动把订单改成 `paid`
- 进入 reviewer 审核台
- 进入 admin 管理后台
- 修改实验配置、DOM 节点表和 Prompt
- 触发不属于自己的自动填报任务

### reviewer 可见页面

reviewer 面向人工审核和纠错，不负责系统配置和支付管理。

页面：

- reviewer 登录页
- 审核任务列表
- 识别结果审核页
- 图片对照查看页
- 固定填空和实验问题处理页
- 审核完成确认页
- 异常任务查看页

reviewer 允许操作：

- 查看分配给自己的审核任务
- 查看对应任务的图片和 AI 识别结果
- 修改 `corrected_json`
- 补充完整模式中的固定填空和实验问题答案
- 标记审核完成
- 将任务推送到自动填报队列
- 查看自己处理过的任务记录

reviewer 禁止操作：

- 查看未分配且无权限的学生任务
- 修改订单金额
- 修改支付状态
- 创建或删除用户
- 修改实验配置
- 修改 DOM 节点表
- 修改 Prompt
- 修改系统级 API Key
- 删除审计日志

### admin 可见页面

admin 是系统管理员，负责配置、订单确认、人员管理和异常处理。

页面：

- admin 登录页
- 管理总览
- 用户管理
- 订单管理(人工收款确认页)
- 任务管理(可分配任务给reviewer)
- 实验配置管理
- DOM 节点表管理
- Prompt 管理
- 自动化任务监控（可失败任务处理）
- 日志页

admin 允许操作：

- 创建、禁用、修改 reviewer 账号
- 查看所有订单和任务
- 在阶段 1 人工收款模式下确认订单 `paid`
- 处理退款或取消订单
- 分配审核任务给 reviewer
- 修改实验配置
- 修改 DOM 节点表
- 修改 Prompt
- 重试失败的自动填报任务
- 查看系统日志和审计日志

admin 也应受到限制：

- 不应该看到明文学校密码
- 不应该直接下载全部学生图片，除非业务确实需要
- 修改关键配置必须记录审计日志
- 确认支付、退款、重试自动提交等高风险动作必须二次确认

### 5.3 后端防越权规则

所有接口必须做服务端权限判断。

基础规则：

1. 未登录不能访问任何任务、订单、图片和配置接口
2. `student` 只能访问 `submissions.user_id = 当前用户 id` 的数据
3. `student` 只能访问自己的 `orders`
4. `reviewer` 只能访问分配给自己的审核任务，或后台允许的待审核队列
5. `reviewer` 不能修改订单和支付状态
6. `admin` 才能确认人工收款、退款、修改配置和管理用户
7. 自动填报前必须检查订单已支付
8. 自动填报前必须检查任务模式和当前状态合法
9. 文件下载必须检查文件所属任务的访问权限
10. 所有高风险操作必须写入 `audit_logs`

后端不应相信这些前端传来的值：

```text
price
payment_status
role
user_id
order_id 所属关系
submission_id 所属关系
```

这些值必须由后端根据当前登录态、数据库记录和业务规则重新校验。

## 6. 核心数据表设计

### 6.1 users

保存用户信息。

```text
id
username
student_no
real_name
hashed_password
role
capabilities
created_at
updated_at
```

字段说明：

- `username`：平台登录账号，当前代码中已存在并唯一；学生用户历史上会用学号作为 username，admin / reviewer 则是后台账号名。
- `student_no`：学生学号，真实学校系统登录账号；admin / reviewer 可为空。
- `real_name`：学校系统同步到的真实姓名，仅用于展示、核对和人工确认，不参与登录。
- `hashed_password`：平台登录密码哈希，不保存学校系统明文密码。

角色建议：

```text
student
reviewer
admin
```

### 6.2 experiments

保存实验项目。

```text
id
name
enabled
config_json
created_at
updated_at
```

`config_json` 用于保存实验对应的提示词、图片编号、填空规则等配置。

### 6.3 dom_mappings

保存学校实验报告系统的 DOM 节点表。

```text
id
experiment_id
version
mapping_json
enabled
created_at
updated_at
```

`mapping_json` 用于保存字段 ID、选择器、填写规则、特殊处理逻辑等。

### 6.4 submissions

保存用户提交的实验任务。

```text
id
user_id
experiment_id
mode
status
payment_status
order_id
price
created_at
updated_at
```

`mode` 示例：

```text
assist
full_review
full_submit
```

说明：

- `assist`：工具辅助模式
- `full_submit`：完整模式，后台人工处理并自动提交

`status` 示例：

```text
created
uploaded
recognizing
recognition_done
reviewing
review_done
automation_pending
automation_running
automation_done
completed
failed
cancelled
```

`payment_status` 示例：

```text
unpaid
paid
refunded
cancelled
```

任务只有在 `payment_status = paid` 后才允许进入识别、审核或自动填报流程。

### 6.5 uploaded_files

保存上传图片记录。

```text
id
submission_id
file_name
storage_key
image_index
mime_type
size
created_at
```

### 6.6 recognition_results

保存 AI 识别结果和纠错结果。

```text
id
submission_id
raw_ai_output
normalized_json
corrected_json
status
created_at
updated_at
```

说明：

- `raw_ai_output`：AI 原始返回
- `normalized_json`：平台解析后的结构化结果
- `corrected_json`：用户或后台人工纠错后的最终结果

### 6.7 automation_jobs

保存自动填报任务。

```text
id
submission_id
status
attempts
started_at
finished_at
error_message
log_json
created_at
updated_at
```

### 6.8 audit_logs

保存关键操作日志。

```text
id
user_id
submission_id
action
payload_json
created_at
```

用于追踪：

- 用户上传了什么
- 谁修改了识别结果
- 谁触发了自动填报
- 自动填报是否提交
- 任务失败原因

## 8. 收费设计

建议按模式和服务深度分层。

### 8.1 工具辅助模式

中低价档。

包含：

- 图片上传
- AI 识别
- 在线纠错
- 用户自行补充固定填空
- 用户自行完成实验问题答案
- 自动登录学校系统
- 自动填报
- 自动提交
- 截图留痕

不包含：

- 人工审核
- 人工纠错
- 人工代查、代改、代填
- 固定填空代写
- 实验问题答案生成

适合：

- 愿意自己纠错和补充固定内容的用户
- 希望平台完成登录、填报和提交的用户
- 对价格比完整模式更敏感的用户

收费方式：

```text
按单次实验收费
按自动填报次数包收费
按会员套餐收费
```

### 8.2 完整模式：人工审核 + 固定内容处理 + 自动填报

中高价档。

包含：

- 工具辅助模式全部能力
- 后台人工纠错
- 固定填空处理
- 实验问题答案生成
- 自动登录学校系统
- 自动填入报告内容
- 自动提交，或提交前等待用户确认
- 截图留痕
- 异常处理

适合：

- 不想自己操作的用户
- 愿意为省时间付费的用户

## 9. 支付与订单系统设计

支付系统的核心原则是：

```text
先选择模式和实验
再生成订单
支付成功后创建或激活任务
任务根据订单中的 mode 决定后续流程
```

用户选择服务时，前端应明确让用户选择：

```text
实验项目
服务模式
是否自动提交
价格
```

服务模式建议用固定枚举值保存：

```text
assist      工具辅助模式
full_submit 完整模式：人工处理并自动提交
```

### 9.1 支付流程

推荐流程：

```text
用户选择实验和服务模式
  |
前端请求后端创建订单
  |
后端根据 experiment_id + mode 计算价格
  |
生成 orders 记录
  |
返回支付二维码或支付链接
  |
用户完成支付
  |
支付平台回调后端 webhook
  |
后端校验回调签名和金额
  |
订单状态改为 paid
  |
创建或激活 submission
  |
任务进入上传/识别/审核/自动填报流程
```

### 9.2 如何区分用户选择了哪种模式

不要靠前端按钮文字判断模式，而是订单和任务里都保存 `mode`。

```text
orders.mode
submissions.mode
```

例如：

```text
mode = assist
```

表示：

- 用户自己纠错
- 用户自己补充固定填空和实验问题
- 平台自动登录、自动填报、自动提交
- 不进入人工审核队列

```text
mode = full_review
```

表示：

- 后台人工纠错
- 后台处理固定填空和实验问题
- 后台自动填报
- 提交前可等待用户确认

```text
mode = full_submit
```

表示：

- 后台人工纠错
- 后台处理固定填空和实验问题
- 后台自动填报
- 后台自动正式提交

### 9.3 推荐数据表

### products

保存可售卖的服务档位。

```text
id
name
mode
price
enabled
created_at
updated_at
```

示例：

```text
工具辅助模式 assist 9.90
完整自动提交 full_submit 29.90
```

### orders

保存用户订单。

```text
id
user_id
experiment_id
product_id
mode
amount
status
payment_provider
provider_trade_no
created_at
paid_at
cancelled_at
```

`status` 示例：

```text
created
pending_payment
paid
cancelled
refunded
```

### payments

保存支付流水。

```text
id
order_id
provider
provider_trade_no
amount
status
raw_callback_json
created_at
updated_at
```

说明：

- `orders` 表表示用户买了什么
- `payments` 表表示支付平台实际发生了什么
- 一个订单通常对应一条支付流水，但拆开后更方便处理回调、退款和异常

### 9.4 支付方式选择

如果前期用户量小，可以先用人工确认收款：

```text
用户扫码付款
上传付款截图或填写转账备注
admin 后台确认订单 paid
系统激活任务
```

阶段 1 推荐先采用人工收款确认。

页面流程：

```text
学生选择实验和服务模式
  |
系统生成订单
  |
学生看到收款码、订单金额、订单编号
  |
学生扫码付款
  |
学生填写付款备注或上传付款截图
  |
订单状态变为 pending_review
  |
admin 在后台核对收款
  |
admin 点击确认收款
  |
orders.status = paid
  |
submissions.payment_status = paid
  |
系统允许任务继续
```

人工收款模式下的订单状态建议：

```text
created
pending_payment
pending_review
paid
cancelled
refunded
```

admin 确认收款时必须校验：

- 订单编号
- 用户学号
- 实验项目
- 服务模式
- 应付金额
- 实收金额
- 付款截图或转账备注

这样开发最快，适合前期验证，但人工成本高。

成熟后可以接入第三方支付：

```text
微信支付
支付宝
第三方聚合支付
```

正式接入第三方支付后，必须以后端支付回调为准，不能以前端“支付成功页面”为准。

### 9.5 防止绕过支付

关键限制：

1. 前端传入的价格不能直接相信
2. 后端根据 `product_id`、`experiment_id`、`mode` 重新计算价格
3. 只有订单 `status = paid` 后，任务才能继续
4. `submissions.order_id` 必须关联已支付订单
5. 支付回调必须校验签名
6. 回调金额必须等于订单金额
7. 同一个订单的支付回调要能幂等处理，重复回调不能重复创建任务

## 10. 安全与合规注意事项

建议原则：

1. 可在后端保存学校账号密码，但是不能暴露给前端和任何用户
2. 保存自动化截图和任务日志
3. 管理员操作必须记录审计日志
4. API Key 不写入前端代码，不提交到仓库
5. 用户上传图片应设置访问权限
6. 对象存储文件应使用私有桶和临时访问链接

权限安全原则：

1. 前端隐藏页面不等于权限控制
2. 每个后端接口都必须检查登录态和角色
3. student 只能访问自己的订单、任务、图片和结果
4. reviewer 只能访问被分配或允许审核的任务
5. admin 的支付确认、退款、配置修改必须写审计日志
6. 任何用户都不能通过修改 URL 查看别人的任务
7. 任何用户都不能通过修改请求体改变自己的 `role`
8. 任何用户都不能通过修改请求体改变订单金额和支付状态
9. 文件接口必须使用鉴权下载，不能暴露永久公开链接
10. 学校密码不能返回给前端，也不能在日志中输出

## 11. 开发路线

### 阶段 1：MVP 工具辅助模式

目标是先把用户自助纠错、自动填报和自动提交流程网站化，但不引入人工代改服务。

功能：

- 用户登录
- 实验选择
- 图片上传
- AI 识别
- 识别结果在线编辑
- 固定填空和实验问题手动填写界面
- Playwright 自动登录学校系统
- 自动填报
- 自动提交
- 截图和日志保存
- 基础任务状态

暂不做：

- 人工审核
- 人工代查、代改、代填
- 固定填空代写
- 实验问题答案生成
- 复杂计费

### 阶段 2：管理后台

功能：

- 实验配置管理
- Prompt 管理
- DOM 节点表管理
- 识别结果查看
- 用户任务管理
- 人工纠错后台

### 阶段 3：完整模式试运行

功能：

- 用户提交完整模式任务
- 后台人工审核
- Playwright 自动填报
- 截图保存
- 日志保存
- 失败状态展示

建议先做到：

```text
后台填好，但不自动最终提交
```

这样可以降低风险。

### 阶段 4：完整模式成熟版

功能：

- 自动提交前用户二次确认
- 自动正式提交
- Playwright 并发控制
- 异常重试
- 验证码处理
- 任务排队
- 通知提醒

### 阶段 5：商业化运营

功能：

- 套餐管理
- 余额 / 次数包
- 优惠码
- 订单系统
- 管理员结算
- 使用统计
- 高峰期任务限流

## 12. 部署建议

初期使用 Docker Compose。

服务组成：

```text
frontend
api
worker
redis
postgres
minio
```

后期如果任务量增大，可以扩展：

```text
多个 worker
独立 Playwright worker

云对象存储

```

## 13. 最终推荐方案

优先采用：

```text
Next.js + FastAPI + PostgreSQL + Redis + Playwright + MinIO
```

推荐上线顺序：

```text
先上线工具辅助模式：用户自助纠错 + 自动填报 + 自动提交
再上线完整模式：后台人工纠错 + 固定填空/实验问题处理
最后完善自动化异常处理、计费和审核后台
```

这样可以最大化复用现有代码，同时把技术风险、账号风险和运营风险逐步拆开。

最重要的是，平台不应该只追求自动化，而应该把流程拆成可审核、可回滚、可追踪、可收费的任务系统。
